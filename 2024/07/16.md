### docker instruction
- a FROM:引数のイメージを適用する
- b RUN:linuxのコマンドを走らせる
- Docker imageのレイヤーは最小限にすることがベストプラクティスとなる。ADD,RUN,COPYがレイヤーを産むのでこれらをうまく扱う必要がある。レイヤーを最小限にする方法はコマンドを`&&`で続けていくこと。&&が続くと見辛くなるので、`/`で改行させ見やすくすることもできる。
- c Dockerfileでは`apt`を使用することが多く`apt-get update && apt-get install <package>`が一般的な書き方となる。`apt-get install <package>`は<package>をインストールすることは容易にわかる。`apt-get update`はパッケージのリストをアップデートしinstallするときに最新のパッケージを扱うようにするもの。」

### dockerレイヤー
- `docker build`のたびにrunコマンドの引数を変更していくとキャッシュされないので、毎回のRUNの時間を要してしまう。
- RUNコマンドはイメージレイヤーが作成されるので、キャッシュ（再利用・軽量）されるためDockerfileをメンテナンスする際はRUNコマンドでレイヤーを積み重ねていき、Dockerfileとして配布する段階になったら`&&`等でレイヤーを少なくしていく。

### RUN, CMDの違い
- RUNはDocker imageを構成していくために必要なコマンドを実行していき、CMDはコンテナを立ち上げた際のデフォルトのコマンドを指定する。RUNはイメージレイヤーを作るが,CMDは作らない。CMDコマンドで`apt-get install <package>`など実行することができるが、パフォーマンスの低下(runで実行すればキャッシュされるので、ビルドが高速。コンテナ起動時に毎回`apt-get install`が実行されては困る）、エラーハンドリングの困難性（コンテナ起動時にパッケージなどをインストールに失敗すると、コンテナが立ち上がらないが、ビルド時にインストールに失敗すればそもそもビルドされないのでそこでエラーが発見できる。）等の理由からRUNにはビルド時に必要なコマンドを記述しCMDにはコンテナ起動時に必要な物だけを記述。

### FROM

- 最小限のものを指定する。大きなものを指定してしまうとその後のRUNを最小限に抑えたとしても意味がないので、最小限のubuntuやaopainなどを最小限のものを指定する。


### その他
- dockerのコンテナはDocker imageから作成され、Docker imageはDockerfileから作成される。Dockerfileにはコンテナを立てた際に実行するコマンドを記述していく。
- dockerイメージはイメージから作成したコンテナからcommitして作成する方とDockerfileから作成する方法がある。チーム開発ではイメージの内容を可視化させるために後者がDocker imageを作成する方法として採用されていることが多い。
